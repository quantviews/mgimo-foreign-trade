# План рефакторинга

## Цель

Снизить операционные риски ETL-пайплайна, упростить сопровождение проекта и сделать изменения в обработке данных более предсказуемыми и тестируемыми.

## Принципы

- Сначала стабилизируем поведение и тестовый контур, потом меняем архитектуру.
- Не смешиваем крупный рефакторинг с изменением бизнес-логики без явной необходимости.
- Каждая итерация должна оставлять проект в рабочем состоянии.

## Приоритеты

- `P0`: защита от потери данных и регрессий.
- `P1`: уменьшение связности и дублирования.
- `P2`: улучшение структуры проекта, документации и DX.

## Итерация 1. Стабилизация и защита от регрессий

Приоритет: `P0`

Оценка сложности: `Средняя`

Оценка объема: `3-5 рабочих дней`

### Задачи

1. Актуализировать тесты для `generate_derived_columns()` и связанных функций под текущую семантику TNVED-кодов.
2. Добавить контрактные тесты для:
   - `china_processor.py`
   - `india_processor.py`
   - `turkey_processor.py`
   - `load_fts_csv.py`
3. Вынести smoke-проверки итогового датасета:
   - допустимые значения `NAPR`
   - наличие обязательных колонок
   - корректный тип `PERIOD`
   - отсутствие пустого результата после merge
4. Сделать безопасную запись DuckDB:
   - сначала писать во временный файл
   - заменять целевой файл только после полного успешного завершения
5. Добавить CI-проверки:
   - запуск `pytest`
   - smoke-check по тестовым данным или минимальным фикстурам
6. Зафиксировать текущие инварианты merge-пайплайна в документации.

### Что не делать в этой итерации

- Не дробить сильно архитектуру.
- Не переписывать процессоры стран на новый интерфейс.
- Не менять доменную логику без тестового покрытия.

### Результат

- Пайплайн защищен от очевидных регрессий.
- Снижен риск потери рабочей базы при неуспешной сборке.
- Есть минимальная автоматическая гарантия корректности ETL.

### Критерии готовности

- Все ключевые тесты проходят локально и в CI.
- `save_to_duckdb()` не удаляет рабочую базу до успешного завершения записи.
- Есть отдельные тесты на каждый источник данных.

## Итерация 2. Выделение доменной логики и сокращение дублирования

Приоритет: `P1`

Оценка сложности: `Высокая`

Оценка объема: `5-8 рабочих дней`

### Задачи

1. Разделить `src/merge_processed_data.py` на модули:
   - `src/core/schema.py`
   - `src/core/tnved.py`
   - `src/core/edizm.py`
   - `src/core/duckdb_writer.py`
   - `src/core/reference_tables.py`
   - `src/pipelines/merge_pipeline.py`
2. Вынести в единый модуль правила:
   - нормализации TNVED
   - генерации `TNVED2`, `TNVED4`, `TNVED6`, `TNVED8`
   - нормализации `EDIZM`
   - специальных кейсов по `KG`, `TONNE`, `BQ`, `БЕККЕРЕЛЬ`
3. Убрать дублирующиеся маппинги единиц измерения из country processors и перевести их на общий слой.
4. Ввести единый контракт country processor:
   - стандартные входы
   - стандартный выходной DataFrame
   - единые post-processing шаги
5. Сократить объем `main()` до orchestration-слоя.

### Что не делать в этой итерации

- Не менять формат итоговой базы без необходимости.
- Не делать полный переход на классы, если функций достаточно.
- Не объединять сразу ETL и сайт в одну абстракцию.

### Результат

- Основная доменная логика сосредоточена в повторно используемых модулях.
- Изменение правил TNVED и EDIZM происходит в одном месте.
- Процессоры стран становятся тоньше и предсказуемее.

### Критерии готовности

- `merge_processed_data.py` сокращен до orchestration-уровня.
- Повторяющаяся логика TNVED и EDIZM удалена из отдельных процессоров.
- Тесты покрывают общий доменный слой, а не только большой интеграционный скрипт.

## Итерация 3. Упорядочивание проекта и эксплуатационная зрелость

Приоритет: `P2`

Оценка сложности: `Средняя`

Оценка объема: `3-6 рабочих дней`

### Задачи

1. Привести документацию к фактической архитектуре:
   - обновить архитектурное описание
   - описать реальный workflow по шагам
   - зафиксировать точки входа для ETL, тестов и публикации
2. Разделить технические документы по уровням:
   - обзор архитектуры
   - доменные правила
   - инструкции по запуску
   - инструкции по публикации
3. Добавить developer guide:
   - как добавить новый источник
   - как обновить маппинги
   - как безопасно менять merge-логику
4. Упорядочить структуру каталогов, где это можно сделать без болезненной миграции.
5. Добавить простые операционные проверки:
   - размер итоговой базы
   - диапазон дат
   - число стран
   - доля unmapped `EDIZM`

### Что не делать в этой итерации

- Не делать крупную миграцию всего репозитория.
- Не трогать Quarto-слой глубже, чем нужно для документации и smoke-check.

### Результат

- У проекта появляется понятная актуальная структура.
- Новому разработчику легче войти в кодовую базу.
- Команда получает базовые эксплуатационные метрики качества данных.

### Критерии готовности

- Документация соответствует реальному состоянию кода.
- Появился единый документ для разработчиков.
- Основные quality checks можно запускать без ручного анализа логов.

## Рекомендуемый порядок выполнения

1. Сначала полностью завершить Итерацию 1.
2. После стабилизации выполнить Итерацию 2 без параллельных больших фич.
3. Затем закрыть Итерацию 3 и зафиксировать новую базовую архитектуру.

## Быстрые победы

- Исправить рассинхрон тестов и логики TNVED.
- Сделать атомарную запись `unified_trade_data.duckdb`.
- Добавить CI-запуск `pytest`.
- Вынести нормализацию `EDIZM` в один модуль до полного рефакторинга.

## Главные риски самого рефакторинга

- Нечаянно изменить доменную семантику TNVED или единиц измерения.
- Перенести код по модулям, но оставить старые скрытые зависимости.
- Начать архитектурный рефакторинг до появления достаточного тестового контура.
