# Модель данных для гармонизированных таблиц таможенной статистики

Этот документ описывает стандартизированную модель данных, используемую для итоговых, гармонизированных наборов данных в директории `data_processed`. Все сырые данные, собранные из разных источников (Китай, Турция, Индия и т.д.), преобразуются в соответствие с этой схемой.

## Схема таблицы `unified_trade_data`

| Название колонки | Тип данных    | Описание                                                                                                                                    | Пример         |
|------------------|---------------|---------------------------------------------------------------------------------------------------------------------------------------------|----------------|
| `NAPR`           | `VARCHAR`     | **Направление торгового потока.** Указывает, является ли торговый поток импортом или экспортом по отношению к Российской Федерации. (`ИМ` = Импорт, `ЭК` = Экспорт). | `ИМ`           |
| `PERIOD`         | `DATE`        | **Отчетный период.** Первый день месяца, за который представлены данные. Всегда хранится в формате `YYYY-MM-DD`.                          | `2023-01-01`   |
| `STRANA`         | `VARCHAR`     | **Страна-партнер.** Двухбуквенный ISO-код страны-партнера (например, `CN` для Китая, `TR` для Турции).                                          | `CN`           |
| `TNVED`          | `VARCHAR`     | **Код ТН ВЭД (Товарная номенклатура внешнеэкономической деятельности).** Полный товарный код в строковом виде. Ведущие нули источника сохраняются; в merge-слое короткие коды при необходимости дополняются справа до 10 знаков. | `0101210000`   |
| `EDIZM`          | `VARCHAR`     | **Дополнительная (supplementary) единица измерения.** Единица измерения для колонки `KOL`, как она указана в исходных данных (например, "Number of item", "kg"). `?` означает отсутствующую или неизвестную единицу. | `Number of item` |
| `EDIZM_ISO`      | `VARCHAR`     | **Код стандартизированной единицы измерения.** Код из справочника `metadata/edizm.csv` для унификации единиц измерения. | `166`          |
| `STOIM`          | `DECIMAL`     | **Стоимость (USD).** Статистическая стоимость товаров в **тыс. долларах** США.                                                                      | `12500.50`     |
| `NETTO`          | `DECIMAL`     | **Вес нетто (кг).** Вес нетто товаров в килограммах.                                                                                         | `450.0`        |
| `KOL`            | `DECIMAL`     | **Количество в дополнительной единице измерения.** Значение в единицах, указанных в колонке `EDIZM`.                                     | `150`          |
| `TNVED8`         | `VARCHAR`     | **Код ТН ВЭД (8 знаков).** Первые 8 знаков кода `TNVED`, представляющие товарную подкатегорию.                                                | `01012100`     |
| `TNVED4`         | `VARCHAR`     | **Код ТН ВЭД (4 знака).** Первые 4 знака кода `TNVED`, представляющие товарную позицию ТН ВЭД.                                                | `0101`         |
| `TNVED6`         | `VARCHAR`     | **Код ТН ВЭД (6 знаков).** Первые 6 знаков кода `TNVED`, представляющие субпозицию ТН ВЭД.                                                   | `010121`       |
| `TNVED2`         | `VARCHAR`     | **Код ТН ВЭД (2 знака).** Первые 2 знака кода `TNVED`, представляющие группу ТН ВЭД.                                                        | `01`           |
| `SOURCE`         | `VARCHAR`     | **Источник ряда.** Маркер происхождения строки: `national`, `comtrade`, `nowcast`.                                                           | `national`     |
| `TYPE`           | `VARCHAR`     | **Тип ряда.** `fact` для фактических данных, `pred` для nowcast-прогнозов.                                                                    | `fact`         |

> Примечание: в Python-пайплайне `PERIOD` хранится как `datetime64[ns]` (с нормализацией времени `00:00:00`), а при записи в DuckDB приводится к типу `DATE`.

## Формат базы данных `db/unified_trade_data.duckdb`

Файл базы: `db/unified_trade_data.duckdb` (DuckDB).

При запуске merge можно указать другой целевой файл через `--output-db-path`; это удобно для candidate-сборок перед публикацией на дашборд.

Содержимое базы:

- Таблица `unified_trade_data` — основная факт-таблица по схеме выше.
- Таблица `tnved_reference` — справочник ТН ВЭД:
  - `TNVED_CODE` (`VARCHAR`)
  - `TNVED_LEVEL` (`INTEGER`, уровни 2/4/6/8/10)
  - `TNVED_NAME` (`VARCHAR`)
  - `TRANSLATED` (`BOOLEAN`)
- Таблица `country_reference` — справочник стран:
  - `STRANA` (`VARCHAR`, ISO2)
  - `STRANA_NAME` (`VARCHAR`)
- View `unified_trade_data_enriched` — join `unified_trade_data` со справочниками + вычисляемый `period_rank`.
- Таблица `fizob_index` (если есть входные `fizob_*` parquet):
  - `STRANA` (`VARCHAR`)
  - `NAPR` (`VARCHAR`)
  - `PERIOD` (`DATE`)
  - `tn_level` (`INTEGER`)
  - `tn_code` (`VARCHAR`)
  - `fizob` (`DOUBLE`)
  - `fizob_bp` (`DOUBLE`)
- View `fizob_index_v` — надстройка над `fizob_index` с вычисляемой колонкой `idx = fizob / fizob_bp`.

## Ключевые концепции и преобразования

- **Гармонизация:** Основная цель скриптов обработки — взять сырые данные в различных форматах и привести их к этой единой, последовательной структуре. Это позволяет проводить унифицированный анализ по разным странам и периодам времени.
- **Нормализация TNVED:** Товарные коды (`TNVED` и его производные) хранятся в виде строк. Ведущие нули не удаляются, потому что они значимы для групп 01-10. В merge-слое короткие коды дополняются нулями справа до 10 знаков, а производные `TNVED2`, `TNVED4`, `TNVED6`, `TNVED8` формируются как префиксы нормализованного кода.
- **Нормализация периода:** Все индикаторы даты из исходных источников (например, `202301`, `Jan-2023`) нормализуются к первому дню соответствующего месяца (`2023-01-01`).
- **Направление торгового потока:** Колонка `NAPR` является ключевой. Ее значение определяется на основе источника. Например, при обработке данных об *экспорте* из Китая в Россию, для России этот поток является *импортом* (`ИМ`). То есть, для национальных данных необходимо **изменить** направление потока: Export -> `ИМ`, Import -> 'ЭК`. 
- **Стандартизация единиц измерения:** Оригинальные единицы измерения из колонки `EDIZM` сопоставляются со стандартным справочником (`metadata/edizm.csv`) для получения унифицированного кода в колонке `EDIZM_ISO`.
- **Происхождение и тип рядов:** Колонка `SOURCE` фиксирует источник (`national`/`comtrade`/`nowcast`), а `TYPE` разделяет фактические значения (`fact`) и прогноз (`pred`).
- **Nowcast только в «дырах»:** при финальном merge (`merge_processed_data` / `merge_pipeline`) строки nowcast (`TYPE='pred'`, `SOURCE='nowcast'`) **отбрасываются**, если для той же **торговой ячейки** уже есть хотя бы одна **фактическая** строка (национальные или Comtrade данные, любой `TYPE`, отличный от `pred`). Ключ совпадения: **`PERIOD`** (первая дата месяца), **`STRANA`**, **`TNVED`** после нормализации к 10 символам (как для сопоставления в `nowcast_compare_2026.R`), **`NAPR`**. Единица измерения `EDIZM` в ключ не входит: наличие факта по той же четверке периода/партнер/код/направление блокирует nowcast целиком по этой ячейке.
- **Физобъемы считаются только по факту:** `src/fizob_queries.R` читает из `unified_trade_data` только строки `TYPE = 'fact'`. Это защищает индексы физических объемов от смешения фактических рядов и nowcast-прогноза.
- **SQL quality gate:** после финального merge можно выполнить проверки из `src/orchestration/checks.py`: обязательные таблицы/колонки, валидные значения `NAPR`/`SOURCE`/`TYPE`, отсутствие `NULL` в `PERIOD` и отсутствие пересечения `pred` с фактом по торговому ключу.
